## 二、验证方法论

### 2.1 总体策略

采用**黑盒二进制对比分析法**，不信任 DiVo 提供的任何脚本或文档，从零开始独立分析两个 `.so` 文件：

- `librtlsim_yuquan.so` — 声称包含白杨 DDR4 MC 的 Verilator 编译产物
- `librtlsim_dramsim.so` — 声称仅含 DramSim 行为模型的对照组

### 2.2 验证维度

| 维度 | 方法 | 期望发现 |
|------|------|---------|
| 文件基本信息 | `file`, `ls -l`, `md5sum`, `sha256sum` | 两个文件不同的二进制 |
| ELF 结构差异 | `readelf -h`, `readelf -S`, `readelf -d` | 白杨版 .text 段更大 |
| 符号表指纹 | `nm -D`, `nm` (无 -D), 交叉对比 | 白杨版独有 YuQuan 符号 |
| 字符串指纹 | `strings -n 6`, `comm` 交叉对比 | 白杨版独有 Chisel 断言 |
| 反汇编验证 | `objdump -d` 抽样 | 确认真实机器码而非填充 |
| 反伪造评估 | 逻辑推演 | 评估伪造可行性和代价 |

### 2.3 关键原则

1. **不信任 DiVo 脚本** — 每个命令手动执行，每个数字独立计算
2. **双向对比** — 不仅看白杨版有什么，还要确认 DramSim 版没有什么
3. **交叉验证** — 同一结论用多种方法确认（如符号 + 字符串 + 反汇编）
4. **记录原始输出** — 保留命令的实际 stdout 作为证据

**符号名称解析**（手工解 mangled）:
```

```
这明确显示了 Verilator 编译的模块层次路径，与白杨 DDR4 MC 的内部结构一致。

**判断**: 白杨版包含 yq_wrapper 符号，DramSim 版完全没有 → 初步确认白杨代码存在于白杨版中。

### 步骤 8：Chisel/FIRRTL 源码断言查找（DiVo 可能遗漏的证据）

**目的**: 搜索 `.scala:` 模式——这是 Chisel 源码断言编译后嵌入二进制的独特指纹，DiVo 脚本未专门搜索此模式。

- **这是 DiVo 脚本未专门搜索的证据模式**

## 五、反伪造评估推理

### 5.1 符号伪造可行性

**评估**: 可行但代价高。理论上可以在 `librtlsim_dramsim.so` 中添加假的 `yq_wrapper` 符号名。但：
- 需要添加的不仅是名称，还包括对应的大小、类型、地址——链接器会检查一致性
- Verilator 生成的符号名带有精确的 C++ mangling（包含参数类型编码），手工生成极易出错
- 符号必须指向真实的代码段，否则 `objdump` 会暴露

### 5.2 .scala 断言伪造可行性

**评估**: 极难。`.scala:行号` 断言包含：
- 精确的源文件名和行号（如 `DFIPhaseCtrl.scala:120`）
- 真实的 Chisel 断言表达式（如 `PopCount(Cat(...)) <= 1.U`）
- 白杨内部信号名（`io.init_process`, `hasRefReqs`, `zqPend`）

要伪造这些，需要：
1. 了解白杨内部源码的行号对应关系
2. 了解 Chisel 的断言语法（`.U` 后缀是 Chisel 特有）
3. 编造出与真实实现一致的信号名

这些字符串嵌在 `.rodata` 段中，是从 Chisel 源码自动生成的——不是手工编写的注释。

### 5.3 文件大小伪造可行性

**评估**: 可行但意义不大。可以填充 3 MB 垃圾数据使文件变大。但：
- `objdump` 会暴露垃圾代码
- 填充数据不会产生新的合法符号
- 无法通过功能测试（kernel 运行结果正确）

### 5.4 综合反伪造评估

**要同时伪造以下所有证据**:
1. 720 个语义正确的 C++ mangled 符号名
2. 8 条精确对应白杨源码行号的 Chisel 断言
3. 3 MB 合理结构的 x86-64 机器码
4. AXI4 适配器的 tag_buf 结构
5. 所有符号与代码段的正确映射
6. 功能测试通过（kernel 结果与 DramSim 一致）

**综合代价**: 等同于实现一个完整的 DDR4 控制器仿真模型。这恰恰就是真正集成白杨的工作量。
